# LLM Wiki

一种用 LLM 构建个人知识库的模式。

这是一份构想文档，设计用于直接粘贴给你的 LLM Agent（例如 OpenAI Codex、Claude Code、OpenCode/Pi 等）。它的目标是传达高层次的核心理念，具体的实现细节由你和你的 Agent 协作完成。

## 核心理念

大多数人使用 LLM 处理文档的方式是 RAG：上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这种方式可行，但 LLM 每次回答问题时都在从零重新发现知识，没有任何积累。如果你问一个需要综合五份文档的深层问题，LLM 每次都要重新拼凑相关片段，什么都没有被建立起来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是这样运作的。

这里的想法不同。它不是在查询时从原始文档检索，而是让 LLM **增量地构建并维护一个持久化的 Wiki** —— 一个结构化、相互链接的 Markdown 文件集合，位于你和原始资料之间。当你添加一份新资料时，LLM 不只是为它建立索引以备后续检索，而是读取它，提取关键信息，并将其整合到现有 Wiki 中 —— 更新实体页面，修订主题摘要，标注新数据与旧观点的矛盾之处，强化或挑战正在演进的综合理解。知识被一次性编译，然后*持续保持最新状态*，而不是在每次查询时重新推导。

这就是关键区别：**Wiki 是一个持久化的、持续复利的产物。** 交叉引用已经就位，矛盾已经被标记，综合已经反映了你读过的一切。每添加一份资料、每提出一个问题，Wiki 就会变得更加丰富。

你从不（或很少）自己编写 Wiki —— LLM 负责编写和维护全部内容。你负责资料的筛选、探索和提出正确的问题；LLM 完成所有繁琐的工作 —— 摘要、交叉引用、归档和记录，这些才是让知识库真正随时间变得有价值的东西。在实践中，我会把 LLM Agent 开在一边，Obsidian 开在另一边。LLM 根据我们的对话进行编辑，我实时浏览结果 —— 跟随链接、查看图谱视图、阅读更新后的页面。Obsidian 是 IDE，LLM 是程序员，Wiki 是代码库。

这一模式可以应用于很多不同的场景，几个例子：

- **个人成长**：追踪自己的目标、健康、心理状态、自我提升 —— 归档日记、文章、播客笔记，逐渐建立起对自己的结构化认知。
- **研究**：在数周或数月内深入某个主题 —— 阅读论文、文章、报告，增量地构建一个带有演进论点的综合 Wiki。
- **读书**：边读边归档每一章，为人物、主题、情节线索及其联系建立页面。读完时你就有了一部丰富的配套 Wiki。想象一下像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 那样的粉丝 Wiki —— 由社区志愿者历经多年构建，涵盖数千个相互链接的角色、地点、事件、语言页面。你可以在阅读时亲手构建类似的东西，由 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：一个由 LLM 维护的内部 Wiki，从 Slack 讨论、会议记录、项目文档、客户通话中持续获取信息，人类可以参与审核更新。Wiki 保持最新，因为 LLM 承担了团队中没有人愿意做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、兴趣深耕** —— 任何你在长期积累知识、希望它有条理而非散乱分布的场景。

## 架构

系统分为三层：

**原始资料层** —— 你精心筛选的原始文档集合。文章、论文、图片、数据文件。这一层是不可变的 —— LLM 只读取，从不修改。这是你的真相来源。

**Wiki 层** —— 一个由 LLM 生成的 Markdown 文件目录。摘要、实体页面、概念页面、对比分析、概述、综合理解。LLM 完全拥有这一层。它创建页面，在新资料到来时更新页面，维护交叉引用，保持一切一致。你读，LLM 写。

**Schema 层** —— 一份文档（例如 Claude Code 的 `CLAUDE.md` 或 Codex 的 `AGENTS.md`），告诉 LLM Wiki 的结构是什么、约定是什么，以及在摄取资料、回答问题或维护 Wiki 时应遵循什么工作流。这是核心配置文件 —— 它让 LLM 成为一个有纪律的 Wiki 维护者，而不是一个通用聊天机器人。你和 LLM 会随着时间的推移，在探索中共同演化这份文档。

## 操作

**摄取（Ingest）。** 你将新资料放入原始集合并告诉 LLM 处理它。一个示例流程：LLM 读取资料，与你讨论关键要点，在 Wiki 中写一篇摘要页面，更新索引，更新 Wiki 中相关的实体和概念页面，并在日志中追加一条记录。一份资料可能会涉及 10-15 个 Wiki 页面。我个人倾向于一次摄取一份资料并全程参与 —— 阅读摘要、查看更新，并引导 LLM 决定强调什么。但你也可以一次批量摄取大量资料，减少人工监督。由你来摸索适合自己风格的工作流，并在 Schema 中记录下来供未来的会话使用。

**查询（Query）。** 你向 Wiki 提问。LLM 搜索相关页面，读取它们，并生成带引用的综合答案。答案的形式可以根据问题不同而变化 —— Markdown 页面、对比表格、幻灯片（Marp）、图表（matplotlib）、画布等。重要的洞见：**好的答案可以作为新页面归档回 Wiki。** 你提出的对比分析、某个发现的联系 —— 这些都有价值，不应该消失在聊天记录里。这样，你的探索就像摄取的资料一样，在知识库中持续复利。

**健检（Lint）。** 定期请 LLM 对 Wiki 进行健康检查。检查：页面之间的矛盾，已被新资料推翻的过时观点，没有入站链接的孤立页面，被提及但缺少独立页面的重要概念，缺失的交叉引用，可以通过网络搜索补充的数据空白。LLM 擅长建议新的探究问题和新的资料来源。这让 Wiki 在不断增长的过程中保持健康。

## 索引与日志

两个特殊文件帮助 LLM（以及你）在 Wiki 增长后进行导航，它们服务于不同的目的：

**index.md** 是内容导向的。它是 Wiki 的目录 —— 每个页面都列有链接、一行摘要，以及可选的元数据（如日期或资料数量）。按类别组织（实体、概念、资料来源等）。LLM 在每次摄取时更新它。回答查询时，LLM 先读取索引找到相关页面，再深入阅读。在中等规模下（约 100 份资料、数百个页面），这种方式出人意料地有效，无需基于嵌入的 RAG 基础设施。

**log.md** 是时间导向的。它是追加式记录，按时间顺序记录发生了什么以及何时发生 —— 摄取、查询、健检。一个实用技巧：如果每条记录以一致的前缀开头（例如 `## [2026-04-02] ingest | 文章标题`），日志就可以用简单的 Unix 工具解析 —— `grep "^## \[" log.md | tail -5` 会给你最近 5 条记录。日志提供了 Wiki 演进的时间线，帮助 LLM 了解最近做了什么。

## 可选：CLI 工具

在某个时候，你可能想构建一些小工具来帮助 LLM 更高效地操作 Wiki。最显而易见的是一个 Wiki 页面的搜索引擎 —— 在小规模下，索引文件已经够用，但随着 Wiki 增长，你需要真正的搜索能力。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个针对 Markdown 文件的本地搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序，全部在本地运行。它既有 CLI（供 LLM 调用 shell 命令），也有 MCP Server（供 LLM 作为原生工具使用）。你也可以自己构建更简单的东西 —— LLM 可以在需要时帮你 vibe-code 出一个朴素的搜索脚本。

## 技巧与建议

- **Obsidian Web Clipper** 是一个浏览器扩展，可将网页文章转换为 Markdown，非常适合快速将资料添加到原始集合中。
- **本地下载图片。** 在 Obsidian 设置 → 文件与链接中，将"附件文件夹路径"设为固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中，搜索"Download"找到"Download attachments for current file"并绑定一个快捷键（例如 Ctrl+Shift+D）。剪藏文章后，按快捷键即可将所有图片下载到本地。这是可选项，但很有用 —— 它让 LLM 可以直接查看和引用图片，而不依赖可能失效的 URL。注意 LLM 无法在一次处理中原生读取带内联图片的 Markdown —— 变通方法是先让 LLM 读文本，再单独查看部分或全部引用的图片以获取额外上下文。稍显繁琐，但效果足够好。
- **Obsidian 的图谱视图**是了解 Wiki 形态的最佳方式 —— 什么与什么相连，哪些页面是枢纽，哪些是孤立点。
- **Marp** 是一种基于 Markdown 的幻灯片格式，Obsidian 有对应插件，可直接从 Wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可对页面的 YAML frontmatter 运行查询。如果你的 LLM 在 Wiki 页面中添加了 frontmatter（标签、日期、资料数量），Dataview 就能生成动态表格和列表。
- Wiki 本质上就是一个 Markdown 文件的 Git 仓库，版本历史、分支和协作功能都是免费获得的。

## 为什么这行得通

维护知识库的繁琐之处不在于阅读或思考，而在于记账：更新交叉引用、保持摘要最新、标注新数据与旧观点的矛盾、在数十个页面间维护一致性。人类放弃 Wiki 是因为维护成本的增长速度超过了它带来的价值。LLM 不会感到厌倦，不会忘记更新某个交叉引用，可以在一次处理中触及 15 个文件。Wiki 能保持被维护，是因为维护的成本趋近于零。

人类的工作是筛选资料、引导分析、提出好问题，以及思考这一切意味着什么。LLM 负责其他所有事情。

这个想法在精神上与 Vannevar Bush 1945 年提出的 Memex 相关 —— 一个带有文档间联想路径的个人精选知识库。Bush 的愿景比万维网后来的走向更接近于此：私密的、主动精选的，文档间的连接与文档本身同等重要。他没能解决的问题是：谁来做维护。LLM 解决了这个问题。

## 说明

这份文档是有意抽象的。它描述的是理念，而非具体实现。确切的目录结构、Schema 约定、页面格式、工具链 —— 这一切都取决于你的领域、你的偏好以及你选择的 LLM。上述所有内容都是可选的、模块化的 —— 取其有用，忽略其余。例如：你的资料可能纯文字，完全不需要图片处理；你的 Wiki 可能足够小，索引文件就是你所需要的全部，不需要搜索引擎；你可能不在乎幻灯片，只想要 Markdown 页面；你可能想要一套完全不同的输出格式。正确的使用方式是将这份文档分享给你的 LLM Agent，共同将其实例化为适合你需求的版本。这份文档唯一的任务是传达这一模式，你的 LLM 能搞定剩下的。
